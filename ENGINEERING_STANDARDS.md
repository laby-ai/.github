# LabY 工程与交付规范

这份规范用来约束 LabY 组织下的项目质量。目标不是增加流程负担，而是避免项目变成难维护的“屎山”，并确保每次交付对真实用户友好、可验证、可继续迭代。

## 1. 基本原则

- 先让用户能顺畅完成任务，再追求内部实现技巧。
- 代码要能被下一个人接手，而不是只有作者当下能看懂。
- 每次改动都要有明确边界：解决什么问题，不解决什么问题。
- 能用现有结构解决的，不额外造一套复杂框架。
- 没有检查方式的改动，默认不算完成。

## 2. 防止屎山代码

### 必须做到

- 单个 PR 聚焦一个目标，避免“顺手重构一大片”。
- 函数、组件、模块命名要表达业务含义，不用 `tmp`、`new2`、`handleDataFinal` 这类名字长期存在。
- 删除无用代码、重复代码和临时调试输出。
- 把复杂逻辑拆成可读的小块，但不要为了抽象而抽象。
- 新增依赖前先说明为什么现有能力不够。
- 修改共享逻辑时，检查调用方和边界情况。

### 明确避免

- 一个文件无限膨胀，什么逻辑都塞进去。
- UI、数据请求、状态处理、业务规则混在一起，无法定位问题。
- 大量复制粘贴，只改几个字段。
- 把“临时方案”写进主流程却没有 TODO、issue 或后续计划。
- 为了赶进度绕过类型检查、lint、测试或错误处理。

## 3. 面向用户交付友好

每次交付要站在使用者角度检查：

- 用户第一眼能看懂当前页面或功能是做什么的吗？
- 出错时有没有可理解的提示，而不是只显示技术报错？
- 加载、空状态、失败状态有没有处理？
- 关键操作有没有反馈，例如保存成功、提交成功、生成中？
- 移动端或小屏幕是否明显破版？
- 文案是否像给真实用户看的，而不是内部变量名或工程说明？

用户交付不只是“功能跑通”。如果用户不知道下一步该做什么，或者失败后不知道怎么恢复，就还没交付完。

## 4. 工程标准化

每个仓库至少应该具备：

- `README.md`：项目目的、安装、运行、检查命令、维护人。
- 明确的启动命令：例如 `npm install`、`npm run dev`、`npm test`。
- 明确的环境变量说明：提供 `.env.example`，不要提交真实 `.env`。
- 基础格式化或 lint：保持团队代码风格一致。
- 清晰目录结构：新增目录前先看已有模式。
- 关键配置不要散落在多个地方却没有说明。

推荐 README 结构：

```text
项目是什么
适合谁使用
怎么本地运行
怎么测试/检查
主要目录说明
环境变量
常见问题
维护人
```

## 5. Git 协作规范

### 分支

不要直接改 `main`。修改已有项目时：

```text
拉最新 main -> 新建分支 -> 修改 -> commit -> push -> Pull Request -> review -> merge
```

分支名建议：

```text
名字/任务简称
feature/功能简称
fix/问题简称
docs/文档简称
```

### Commit

Commit message 要能看懂，不要只写 `update`、`fix`、`change`。

推荐：

```text
fix upload error message
add onboarding guide
update project readme
```

### Pull Request

PR 必须写清楚：

- 改了什么
- 为什么改
- 怎么检查
- 风险和后续工作

PR 应该尽量小。一个 PR 同时改 UI、后端、配置、文档、重构时，很难 review，也很难回滚。

## 6. 测试与检查

不同改动至少对应一种检查方式：

| 改动类型 | 最低检查要求 |
| --- | --- |
| 文档 | 链接能打开，步骤能按顺序读懂。 |
| UI | 关键页面手动打开，检查正常、空状态、错误状态。 |
| 业务逻辑 | 覆盖正常路径和至少一个边界情况。 |
| API / 数据 | 检查请求、响应、错误处理和数据兼容性。 |
| 配置 / 部署 | 说明影响范围，验证启动或构建不破。 |
| 安全相关 | 检查 token、权限、输入输出边界。 |

如果仓库有测试命令，PR 里要写运行结果：

```text
npm test
npm run lint
npm run build
```

如果没有自动测试，也要写手动检查：

```text
手动打开 /settings 页面，修改用户名，保存后刷新仍然保留。
```

## 7. Review 检查清单

Review 时优先看这些问题：

- 这个 PR 是否真的解决了 issue 或任务？
- 是否直接改了 `main` 或绕过了 PR 流程？
- 是否引入重复代码、临时变量、难懂命名或过度抽象？
- 用户失败、空状态、加载状态是否处理？
- 是否提交了敏感信息、个人文件或无关生成文件？
- README、环境变量、操作说明是否需要同步更新？
- 是否有足够的检查证据？

## 8. 合并标准

满足这些条件再合并：

- PR 范围清楚，不混入无关改动。
- 作者说明了验证方式。
- 关键风险已经 review 过。
- 文档或 README 已同步更新。
- 没有泄露 secret、token、`.env` 或个人文件。

如果不确定，就先不合并，留 comment 或开 issue。

## 9. 发现屎山苗头时怎么处理

不要一上来大重构。先做三步：

1. 在 issue 里描述具体问题：哪里难维护，影响什么。
2. 标出最小可改范围：先拆哪个文件、先补哪个测试、先删哪段重复逻辑。
3. 用小 PR 逐步修，不要一次性重写整个模块。

可接受的修复顺序：

```text
补 README / 检查方式 -> 删除无用代码 -> 拆重复逻辑 -> 增加测试 -> 再做结构调整
```

## 10. 新项目立项前检查

新建仓库或新模块前，先回答：

- 这个项目服务谁？
- 第一个可交付结果是什么？
- 怎么判断用户能用？
- 需要 public 还是 private？
- 谁维护？
- 运行和检查命令是什么？

回答不清楚时，先写一个 issue 或 README 草稿，不急着堆代码。
