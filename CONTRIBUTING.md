# Contributing

This organization uses issues and pull requests as the default collaboration flow.

## Before Starting

- Open or claim an issue when the work affects shared code or decisions.
- Keep the scope small and name the expected outcome clearly.
- Ask for repository maintainer input before changing public APIs, deployment behavior, or data contracts.

## Pull Requests

A pull request should include:

- what changed
- why it changed
- how it was verified
- any follow-up work or known limitations

Prefer focused changes over broad refactors. If a cleanup is useful but not required, open a separate issue or pull request.

## Reviews

Review for correctness, maintainability, security, and operational impact. Comments should be specific enough for the author to act on.
